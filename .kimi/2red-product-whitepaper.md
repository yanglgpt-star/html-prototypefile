# 产品白皮书维护 Skill

## 用途
将 PRD 的增量变更合并到产品白皮书中,保持白皮书为全量功能文档。

## 触发条件
当用户要求:
- "将 PRD 合并到白皮书"
- "更新产品白皮书"
- "基于 PRD 生成白皮书"
- "白皮书增量更新"

## 核心原则
1. **白皮书按功能模块逻辑组织**,不按版本时间线
2. **尽量保留 PRD 原文**,保持描述详细程度一致
3. **白皮书文件名需体现版本号**,格式: `产品白皮书_v[版本号].md`
4. **长文档需逐模块合并**并等待用户确认
5. **合并前必须校验**功能模块是否对应,不对应则警告停止
6. **功能描述必须详细**,不能简单概括,需包含完整的交互细节
7. **逐段确认机制**,每完成一段后等待用户确认再继续
8. **提炼用户修改要求**,应用到后续所有段落的编写中

## 执行流程

### 阶段 0: 判断模式
检查白皮书文件是否存在:
- **不存在** → 进入"初版生成模式"(阶段 1)
- **存在** → 进入"增量合并模式"(阶段 2)

### 阶段 1: 初版生成模式
当白皮书文件不存在时:

1. **读取 PRD 文档**
2. **提取核心功能模块**
3. **生成白皮书结构**:
   - 文档头部(版本、更新日期、覆盖范围)
   - 修改日志
   - 功能概述
   - 各功能模块详细内容
   - 功能依赖关系
4. **保存文件**到 `产品需求/` 目录,文件名格式: `[产品名]产品白皮书_v[版本号].md`
5. **告知用户**白皮书已生成

### 阶段 2: 增量合并模式
当白皮书文件已存在时:

#### 2.1 读取文档
- 读取现有白皮书
- 读取新版本 PRD

#### 2.2 模块对应校验
- 提取 PRD 中的功能模块列表
- 提取白皮书中的功能模块列表
- **对比模块结构**:
  - 如果 PRD 包含白皮书中不存在的模块 → 标记为"新增"
  - 如果 PRD 中的模块描述与白皮书不同 → 标记为"修改"
  - 如果白皮书中的模块在 PRD 中不存在 → 标记为"删除"(需特别注意)

#### 2.3 合并策略确认
向用户展示变更摘要:
```
检测到以下变更:
- 新增模块: [模块A, 模块B]
- 修改模块: [模块C]
- 删除模块: [模块D] ⚠️

是否继续合并?
```

#### 2.4 执行合并
根据变更类型执行不同策略:

**新增模块**:
- 在白皮书中找到合适的插入位置(按功能逻辑分组)
- 插入完整的模块内容
- 保持 PRD 原文描述

**修改模块**:
- 定位白皮书中对应的模块章节
- 精准替换修改的部分
- 保持未修改部分不变

**删除模块**:
- ⚠️ **二次确认**:"确定要删除 [模块D] 吗?此操作不可恢复"
- 用户确认后才执行删除
- 精准删除,不误删其他内容

#### 2.5 更新元数据
- 更新版本号(根据变更类型决定主版本/次版本/修订号)
- 更新"最后更新"日期
- 在"修改日志"中添加本次变更记录

#### 2.6 分段确认机制
如果 PRD 或白皮书超过 3000 字:
- 逐模块进行合并
- 每完成一个模块后暂停
- 向用户展示合并结果
- 等待用户确认后继续下一个模块

**重要：分段确认流程**
1. **逐段编写**：每完成一个功能模块的详细描述后，立即停止
2. **展示内容**：向用户展示刚完成的段落内容
3. **等待确认**：明确询问用户"请确认是否符合要求，我继续编写下一段"
4. **提炼修改**：如果用户提出修改意见，必须：
   - 立即修改当前段落
   - 提炼用户的修改要求（例如：需要更详细的时间说明、需要补充异常处理等）
   - 将提炼的要求应用到后续所有段落的编写中
5. **继续编写**：用户确认后才继续下一个模块

**分段确认示例**：
```
助手：我已完成"狼人击杀功能"的详细描述，包含了触发时机、界面元素、操作流程、倒计时机制、异常处理等完整内容。请确认是否符合要求，我继续编写下一段"预言家查验功能"。

用户：确认，继续

助手：好的，我继续编写"预言家查验功能"...
```

**用户修改示例**：
```
用户：倒计时部分需要更详细，要说明剩余多少秒时有什么提醒

助手：好的，我已修改"狼人击杀功能"中的倒计时部分，增加了：
- 剩余30秒：提示条文字变为橙色
- 剩余10秒：倒计时数字开始闪烁
- 剩余5秒：播放"滴答"音效

我会在后续所有功能模块中都按照这个标准详细描述倒计时提醒机制。现在继续编写"预言家查验功能"...
```

### 阶段 3: 完成
- 保存更新后的白皮书
- 向用户报告:
  - 新版本号
  - 变更摘要
  - 文件保存路径

## 注意事项

### 功能描述详细标准
每个功能模块必须包含以下完整内容,不能简单概括:

#### 必需章节
1. **触发时机与系统提示**
   - 什么时候触发该功能
   - 用户收到什么样的提示（文字、语音、视觉）
   - 提示的具体内容和样式

2. **界面状态与元素**
   - 界面布局和元素位置
   - 各元素的尺寸、颜色、样式
   - 不同状态下的界面变化（可用/禁用/选中等）

3. **操作方式与交互反馈**
   - 用户怎么操作（点击哪里、如何表达意图）
   - 每个操作的即时反馈（震动、动画、音效）
   - 操作流程的每一步详细说明
   - 确认弹窗的内容和按钮

4. **时间控制与倒计时**
   - 是否有倒计时
   - 倒计时多长时间
   - 倒计时什么时候开始/结束
   - 倒计时的显示位置和样式
   - 倒计时提醒机制（剩余多少秒时提醒）
   - 超时后的处理逻辑

5. **规则与限制**
   - 使用条件和限制
   - 目标选择限制
   - 次数限制
   - 特殊规则说明

6. **异常处理**
   - 网络异常时的处理
   - 用户掉线时的处理
   - 操作失败时的提示和重试机制
   - 特殊情况的处理逻辑

7. **结果展示**
   - 结果在什么时候显示
   - 显示的具体样式（动画、文案、颜色）
   - 结果对游戏状态的影响
   - 其他玩家看到的内容

8. **数据规则**
   - 数据记录内容
   - 数据的优先级规则
   - 数据的判定逻辑
   - 数据的可见性

#### 描述标准示例
❌ **错误示例**（过于简单）:
```
狼人击杀
- 狼人可在夜晚阶段互相看到身份
- 通过点击或投票选择击杀目标
- 需达成一致意见（多数或全票）
- 击杀结果在白天公布
```

✅ **正确示例**（详细完整）:
```
### 狼人击杀功能

**位置**：夜晚阶段 -> 狼人行动环节
**目标**：狼人协商并选择击杀目标

#### 触发时机与系统提示
- 游戏进入夜晚阶段
- 系统语音播报："天黑请闭眼"
- 所有玩家屏幕变暗（透明度80%黑色遮罩）
- 系统语音播报："狼人请睁眼"
- 狼人玩家屏幕遮罩消失，恢复正常亮度
- 屏幕顶部显示红色提示条："请选择今晚要击杀的目标"
- 倒计时开始：屏幕顶部中央显示"60"（红色数字，字号32px）

#### 界面状态与元素
- 狼人头像：金色边框 + 队友标识（右上角红色狼爪图标）
- 可击杀目标：头像正常显示，可点击
- 已出局玩家：头像置灰，不可点击
- 底部居中显示"确认击杀"按钮（宽度200px，高度48px）
- 按钮默认状态：置灰不可点击

#### 操作方式与交互反馈
1. 狼人点击任意非狼人玩家头像
2. 点击反馈：震动反馈（50ms）
3. 目标头像出现红色击杀标记
4. 实时票数统计显示在头像右上角
5. 点击"确认击杀"按钮提交

[... 继续详细描述其他章节 ...]
```

### 版本号规则
- **主版本号**(X.0.0): 重大功能重构或架构变更
- **次版本号**(x.Y.0): 新增功能模块
- **修订号**(x.y.Z): 功能细节修改或bug修复

### 安全机制
1. **模块不对应时必须警告**并停止合并
2. **删除操作需二次确认**避免误删
3. **长文档分段处理**避免一次性修改过多内容
4. **保留 PRD 原文**避免信息丢失

## 示例对话

**用户**: "将这个 PRD 合并到充电白皮书中"

**助手**:
1. 读取白皮书和 PRD
2. 检测到新增3个模块、修改1个模块
3. 展示变更摘要并询问确认
4. 用户确认后执行合并
5. 更新版本号为 v1.1.0
6. 报告完成

---

**重要**: 每次合并都要在白皮书的"修改日志"中记录变更,方便追溯历史。

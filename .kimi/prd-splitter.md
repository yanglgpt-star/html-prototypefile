# PRD Splitter

接收一份完整 PRD，将其拆分为两份文档：
- **PRD精简**：保留"为什么做、做什么、谁来决策"的对齐层，面向团队
- **Spec**：提取"怎么做、边界在哪、怎么验收"的执行层，面向 AI

---

## 核心原则

PRD 和 Spec 的本质区别是**面向对象不同**：

- PRD 写给人看，依赖读者的专业判断来补全细节
- Spec 写给 AI 执行，不能留任何需要猜测的空间

拆分不是复制粘贴，而是**重新提炼**：PRD 里"好看一点""功能完整"这类描述，在 Spec 里必须转化为可验证的具体标准。

---

## 第一步：读懂原 PRD

读完后，在动笔前先完成两件事：

1. **列出需求点清单**：原 PRD 有几个独立功能点？每个需要生成一份 Spec
2. **标记模糊项**：哪些描述在 PRD 里是模糊的（"好看""完整""合理"），需要在 Spec 里转化或标注"待确认"

---

## 第二步：生成 PRD精简

**每节的提炼要点：**

### 背景与目标

- 保留"为什么"的论述，压缩掉实现细节
- 目标必须带指标（如果原 PRD 有的话），没有指标的目标写"【待补充量化指标】"

### 范围决策

- "不做什么"和"做什么"同等重要，必须都写出来
- 有时原 PRD 没有显式写"不做"，需要根据功能边界反推并补充

### 关键决策记录

- 从评审记录、功能备注、"暂不"/"后续"等措辞中提取决策
- 格式：【决策】xxx 【原因】xxx

### 待确认事项

- 原 PRD 中标注"待确认"的内容直接搬过来
- 原 PRD 中模糊但没有标注的内容，提炼后也放这里，并注明"原 PRD 未明确，建议确认"

---

## 第三步：生成 Spec

**每个独立功能点生成一份 Spec**

### 判断需要几份 Spec

原则：**一个 Spec 对应一个可独立交付、独立验收的任务单元**

| 情况 | 处理方式 |
|-----|---------|
| 前端页面 + 后端接口 | 通常拆成独立 Spec（前端/后端各一份），除非强耦合 |
| 多个页面但同一功能流 | 可以合并为一份，但 I/O 按页面分开描述 |
| 后台配置 + 前台展示 | 拆成两份（运营配置 Spec + 用户侧 Spec） |

### 六个字段的提炼规则

**目标（Goal）**：

- 不写"做一个xx功能"，写"用户能通过xxx完成xxx，达到xxx"
- 必须包含：谁 + 怎么做 + 达成什么

**边界（Scope）**：

- 从原 PRD 的功能点拆分表格 + 备注中提取
- 明确写"本次不包含"，防止 AI 过度实现

**输入输出（I/O）**：

- 字段级别的精度：字段名、类型、长度、必填、校验规则
- 输出覆盖所有状态：成功 / 失败 / 网络异常 / 空状态
- 如果原 PRD 只写了"登录成功跳转"，需要补全：跳转到哪个路径、携带什么参数

**约束（Constraints）**：

- 技术约束：使用什么框架、组件库、接口规范
- 安全约束：数据存储方式、权限控制
- 设计约束：对应设计稿的哪个页面（标注蓝湖链接或页面名）

**验收标准（Acceptance Criteria）**：

- 这是最关键的转化环节：把 PRD 里模糊的描述变成可操作的检查项
- 每条都必须能用"打开页面/调接口"来验证，不允许出现"体验良好""显示正确"这类词
- 原 PRD 里每一条功能描述，至少对应一条验收标准

**转化示例**：

- PRD写："登录成功后跳转" → Spec写："✅ 输入正确邮箱密码点击登录，3秒内跳转到 /dashboard"
- PRD写："显示错误提示" → Spec写："✅ 输入错误密码提交后，按钮下方显示红色文字'邮箱或密码错误'，按钮恢复可点击"
- PRD写："记住我功能" → Spec写："✅ 勾选记住我后，token 存储在 httpOnly cookie 中，关闭浏览器重新打开仍保持登录状态"

**非目标（Non-Goals）**：

- 来自两个来源：①原 PRD 明确说不做的 ②容易被误解为"应该有"但其实本次不做的
- 写非目标是为了防止 AI 顺手加功能

---

## 第四步：输出前检查

输出两份文档前，做一致性检查：

**PRD精简检查**：

- 每个"待确认"是否都有对应的负责人？（没有就写"待指定"）
- "不做什么"是否有覆盖所有容易被误解为"应该有"的功能？
- 关键决策是否都有记录原因，而不只是结论？

**Spec检查**：

- 验收标准里有没有出现"正常""正确""合理""良好"这类不可验证的词？有则替换掉
- I/O 是否覆盖了所有异常情况（不只是成功路径）？
- 非目标是否明确排除了"AI 最容易顺手做的功能"？

**两份文档之间的检查**：

- PRD精简里的每个需求点，Spec 里是否都有对应的文档？
- PRD精简里标注"待确认"的项，Spec 里是否也标注了"待确认"，而不是自行填了一个答案？

---

## 输出格式

两份文档均用 **Markdown** 格式输出：

```
PRD精简_[功能名称].md    ← 面向团队对齐
specs/[功能名称]-[需求点].md  ← 面向 AI 执行
```

# PRD Writer

根据用户提供的原始素材，按标准流程输出完整、可交付技术团队的PRD文档。

---

## 第一步：读懂原始素材

在动笔前，先把所有素材消化完，不要边读边写。

**读取顺序**（按信息密度从高到低）：

1. AI智能纪要 / 结构化摘要 → 优先提取，信息密度高、结构清晰
2. 会议文字记录 / 逐字稿 → 补充细节、争议点、未决事项
3. UI 设计稿 / 截图 → 逐页分析交互细节（见第三步）
4. 其他补充文档（竞品截图、数据报告等）

**读素材时，同步完成四层信息提取**：

| 层次 | 提取内容 | 写法 |
|-----|---------|------|
| 业务背景 | 为什么做？当前有什么问题？ | 用于「三：需求背景」 |
| 核心规则 | 关键业务逻辑、计算公式、判断条件 | 用于「五：需求详情」 |
| 系统边界 | 本期做什么、明确不做什么 | 用于需求点拆分 + 备注 |
| 待确认项 | 会议中争议未决、说了"后续再定"的内容 | 在PRD中显式标注"待确认" |

> **关键原则**：所有信息来自素材，不自行发明逻辑。遇到素材中没有定论的事项，写"待XX确认"，不能伪装成已确定。

---

## 第二步：生成初稿

按 `你定义的 PRD 框架.md` 的骨架逐节填充。以下是每节的撰写要点：

以我要求的 PRD 框架为例：

### 一：前言

- 修订记录：版本号从 V1.0 开始，操作时间填今天日期
- 相关文档：把原始素材的名称和简要说明记录在此

### 二：项目成员

- 从素材中提到的人名推断角色，不确定的留空，不猜测

### 三：需求背景

**问题现状**写三个维度：

- 用户/业务侧：当前存在什么痛点
- 系统侧：技术/产品层面的缺失
- 竞品对比（如素材中有提及）

**需求目标**：明确写出要解决什么问题、期望达成什么效果、有无数据指标目标

### 四：需求概述

**需求点拆分表格**：

- 按优先级（P0/P1/P2）拆分，区分前端/服务端/后台/运营等不同视角
- 本期不做的项目也要列出来，备注"延后"

**流程图**：

- 用文字流程图（ASCII风格）描述核心业务逻辑和页面跳转
- 复杂流程拆成多个流程图（如：核心系统流程图 + 用户侧页面流转图 + 状态流转图）
- 格式示例：

```
入口 → 【页面A】
    │
    ├── 条件满足 → 【页面B】→ 结果
    └── 条件不满足 → 提示文案
```

### 五：需求详情

每个需求点单独一节，**按受众分层描述**：

- **后台/运营需求**：配置项字段名、类型、说明（用表格）、字段之间的联动关系
- **服务端需求**：校验逻辑、计算公式（带示例数字验算）、订单/状态处理规则、异常处理
- **前端需求**：见第三步，页面逐元素描述

**结构层级**：`需求点N` → `N.x 子节` → `#### 细项`，保持编号一致。

### 七：埋点

每个关键用户行为一行，字段：站点、事件中文名、事件英文名（snake_case）、触发时机、属性中文名、属性英文名、属性值类型

### 八：风控

- 是否涉及权益发放、资金变动、用户隐私
- 重复申请/操作的防护机制
- 日志留存要求

### 九：A/B TEST

- 若本期全量上线：写明"直接全量上线，不做A/B实验"，并给出后续可观测的假设性实验方案
- 若做实验：写清实验组/对照组差异、核心度量指标、处理预案

## 第三步：结合 UI 稿补充前端交互细节

有 UI 稿时，按以下逻辑逐页分析，每个页面作为一个独立需求点（或子节）：

**分析每个页面的顺序**：

1. **识别页面类型**：这是列表页、详情页、弹窗、还是海报？
2. **清点所有 UI 元素**：从上到下、从左到右逐一列举
3. **为每个元素定义**：
   - 展示条件（什么情况下显示/隐藏）
   - 数据来源（字段名是什么、从哪里取）
   - 交互行为（点击/滑动/输入后发生什么）
4. **识别状态变体**：同一页面在不同状态下（如：处理中/成功/失败/空状态）的差异逐一描述
5. **独立模块单独处理**：底部固定按钮、浮层弹窗、分享海报等独立作为需求点

**常见需要专项说明的细节**：

- 空状态：无数据时展示什么
- 超长文本截断规则
- 按钮的显隐条件（尤其是操作型按钮）
- Toast / 弹窗的触发时机和文案
- 跳转目标页面

---

## 第四步：迭代补全与一致性检查

每次增补需求后，检查三处保持一致：

1. `四：需求概述 → 需求点拆分表格`：有没有新增行？
2. `四：需求概述 → 流程图`：有没有新增节点/分支？
3. `五：需求详情`：对应需求点是否已有完整描述？

## 关键撰写原则（贯穿全程）

| 原则 | 具体做法 |
|-----|---------|
| **有据可依** | 每条规则都来自素材，不凭感觉补逻辑 |
| **待定项显式标注** | 文案、入口位置、接口字段未确认的，写"待运营/设计/开发确认"，不留空也不自行填写 |
| **受众分层** | 后台配置（运营视角）、校验规则（服务端视角）、页面交互（前端视角）分开描述，不混写 |
| **边界清晰** | 本期做什么、不做什么，都要明确写出来 |
| **可追溯** | 关键决策尽量能对应回素材中的具体讨论 |

## 输出格式

- 使用 **Markdown** 格式输出
- 文件命名建议：`PRD_[功能名称].md`
- 保存至用户指定目录

## 文档说明

### PRD精简_代储代销.md

**用途**：面向产品经理、项目经理、业务方进行需求对齐

**内容**：
- 背景与目标（为什么做）
- 范围决策（做什么、不做什么）
- 关键决策记录
- 待确认事项
- 角色与权限

### Spec 文档（specs/目录）

**用途**：面向开发工程师、测试工程师进行功能实现

**每份 Spec 包含**：
- Goal：功能目标（谁 + 怎么做 + 达成什么）
- Scope：边界范围（包含/不包含）
- I/O：输入输出（接口字段、校验规则、异常情况）
- Constraints：约束条件（技术/业务/设计约束）
- Acceptance Criteria：验收标准（可验证的检查项）
- Non-Goals：非目标（防止过度实现）

## 关键待确认事项汇总

1. **运营后台配置**：费率、仓库等配置缺少运营后台入口
2. **行情系统对接**：行情数据获取方式和接口规范
3. **平安钱包支付**：拨付确认后的具体支付接口调用
4. **服务费率计算单位**：PRD中"‱"（万分之）与示例"1%"存在矛盾
5. **分账触发时效**：清结算中心仅展示，触发时效性要求
6. **票据中心功能**：开具发票、下载功能的实现方式
7. **出库指令触发**：货款到账后自动触发还是人工发布

详细待确认事项见 `PRD精简_代储代销.md` 中的「待确认事项」章节。

## 使用建议

### 团队对齐阶段
1. 产品/业务方阅读 `PRD精简_代储代销.md`
2. 确认「范围决策」中的"本期不包含"是否准确
3. 逐一确认「待确认事项」，补充缺失信息
4. 确认「关键决策记录"，确保决策原因被理解

### 开发实施阶段
1. 开发工程师阅读对应功能的 Spec 文档
2. 按照 Spec 中的 I/O 定义接口契约
3. 按照 Spec 中的 Constraints 实现约束
4. 按照 Spec 中的 Acceptance Criteria 进行自测
5. 测试工程师按照 Acceptance Criteria 编写测试用例

### 需求变更时
1. 更新 `PRD精简_代储代销.md` 中的相关章节
2. 更新对应 Spec 文档的 Goal/Scope/I/O/Constraints
3. 检查 Acceptance Criteria 是否需要同步更新
4. 在 PRD精简的「更新日志」中记录变更
